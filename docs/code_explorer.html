Patterns

Three good patterns and two realistic issues. Each card is written in three layers so students can move from the obvious behavior to the design consequence.

Good Pattern 1 - Encapsulated behavior

Square keeps state and movement logic together, which makes the object easier to reason about than a pile of separate globals.

BasicsThe class holds x, y, size, speed, color, and motion methods in one place.
Engineering InsightChase, flee, update, and draw can evolve together because they share the same object state.
Architecture InsightThis is a small example of object-oriented design: data and behavior stay close to each other.

Good Pattern 2 - Clear frame order

The main loop follows a stable sequence: poll events, clear, handle lifetime changes, update, draw, present.

BasicsStudents can trace the same steps every frame without guessing what comes next.
Engineering InsightA fixed update order reduces accidental bugs when the simulation grows.
Architecture InsightIt separates input, state mutation, and rendering, which is a core game-loop pattern.

Good Pattern 3 - Config constants

Screen size, radii, FPS, and motion bounds live at the top of the file, so the simulation has a single visible configuration area.

BasicsChanging values like 800, 600, or 60 does not require hunting through the whole file.
Engineering InsightMagic values are reduced, so tuning the simulation is safer.
Architecture InsightCentral constants act like a simple configuration layer for the module.

Potential Issue 1 - Quadratic neighbor scans

Each square checks every other square in both chase and flee, so the work grows quickly as the number of objects increases.

BasicsWith 20 squares the cost is fine, but the loop would slow down if the count increased a lot.
Engineering InsightThis is the main scalability risk in the current design.
Architecture InsightSpatial buckets or hashing would let the program ask fewer pair questions per frame.

Potential Issue 2 - In-place respawn logic

Expired squares are replaced directly inside the list during the loop, which can make a frame depend on where the replacement happens.

BasicsEarlier squares and later squares do not always see the same world state.
Engineering InsightThat makes behavior a little harder to test because the simulation is partly order dependent.
Architecture InsightA separate spawn pass would make the frame more predictable.

Potential Issue 3 - Import side effects

pygame.init() runs as soon as the module is imported, which is convenient for a script but awkward if the file is ever reused.

BasicsImporting the file does more than define functions and classes.
Engineering InsightThat makes automated tests and reuse a little harder.
Architecture InsightStartup code is usually easier to manage when it stays inside main.

Code Review

Five review points, ordered from correctness to maintainability. Click a card on the left to inspect the detail panel on the right.

Performance

The current design is lightweight for 20 objects, but the neighbor scan pattern will become the bottleneck first if the square count grows.

Wins

In-place list reuse keeps the object count steady, the math per square is small, and the frame uses a fixed 60 FPS cap rather than busy waiting.

Risks

The pairwise chase and flee scans are O(n²), the FPS text is re-rendered each frame, and every square performs several math operations per update.

4 / 6Verdict: fine for a classroom demo, but scaling will hurt unless neighbor checks are reduced.
